# 4MicTDM 采集全零（频谱为空）诊断与修复

日期: 2026-06-09

## 现象演进
1. 最初「频谱为空」: WAV 头未封口 —— `wave` 模块只在 `close()` 回填 RIFF/data size，
   Windows 下 Ctrl+C 无法中断阻塞 `recv()`，强杀进程 → `finally` 没跑 → 头里长度=0 → 读出 0 采样。
   **已修**: `recv_server.py` 加 socket 超时(可中断)、周期 `patch_wav_sizes()` 封口、SIGINT/SIGTERM 优雅退出。
2. 修头后 WAV 合法(29s/4ch)，但**4 通道样本精确全 0** —— 不是噪声，是数字静音。
   固件数据通路无补零 bug（i2s_read 直接发），故是 PCM1865 没把数据吐到 DOUT。
3. I2C 正常: 启动扫描发现 `0x4A`，所有寄存器写均 ACK（芯片是活的）。

## 根因
PCM1865 **无 MCLK 时没有系统时钟**。原方案设想「从机 + 内部 PLL 以 BCK 为参考造系统时钟」，
但该模式 Linux 内核驱动都未实现，需手填整套 PLL 寄存器(P/R/J/D + SCK/DSP/ADC 分频)；
未正确配置 → PLL 不锁 → ADC 无时钟 → 数字输出恒 0。**精确全零正是锁不到时钟的表现。**

## 权威寄存器图（torvalds/linux pcm186x.h）
- CLK_CTRL 0x20: bit7/6 SCK_XI_SEL, bit5 SCK_SRC_PLL, bit4 MST_MODE, bit3 ADC_SRC_PLL,
  bit2 DSP2_SRC_PLL, bit1 DSP1_SRC_PLL, bit0 CLKDET_EN
- PLL: CTRL 0x28(bit4 LOCK只读/bit1 REF_SEL/bit0 EN), P 0x29, R 0x2A, J 0x2B, D 0x2C/0x2D
- 分频: PLL_SCK_DIV 0x25, DSP1 0x21, DSP2 0x22, ADC 0x23, BCK 0x26, LRK 0x27
- 状态(只读): DEVICE 0x72, FSAMPLE 0x73, DIV 0x74,
  CLK_STATUS 0x75(bit6 LRCKHLT/5 BCKHLT/4 SCKHLT/2 LRCKERR/1 BCKERR/0 SCKERR), SUPPLY 0x78

## 诊断手段（已加入固件）
`pcm1865_dump_status()` 在 I2S 使能后回读 0x20/0x28/0x72-0x75/0x78 并解码打印。
判读:
- SCKHLT=1 → 无系统时钟（缺 MCLK 或 PLL 没锁）← 预期就是这个
- BCKHLT/LRCKHLT=1 → ESP32 没送出 BCK/LRCK（I2S 或接线问题）
- SUPPLY 某位=0 → 对应电源轨异常

## 修复（固件已实现，config.h 开关 USE_MCLK_OUTPUT）
- **Path A（默认=1，推荐）**: ESP32 额外输出 256fs MCLK (GPIO10) → PCM1865 SCK/XI 脚；
  CLK_CTRL=0x01(CLKDET_EN) 自动检测。需**多接 1 根线**。业界标准做法。
- **Path B（=0，免线）**: BCK-PLL，pcm1865.c 留骨架，P/R/J/D 待按数据手册填+逻辑分析仪验证。

## 下一步
1. 先烧当前固件，贴启动 `=== PCM1865 状态 ===` 日志确认 SCKHLT。
2. 走 Path A: GPIO10 → PCM1865 SCK 脚接线，烧录(USE_MCLK_OUTPUT=1)，看 RMS 是否非 0。
3. 若无法接线再走 Path B，届时据数据手册算 PLL 系数。
